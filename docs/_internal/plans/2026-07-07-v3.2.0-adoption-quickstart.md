# v3.2.0 — Adoption & No-Docker Quickstart

**Status:** approved (Codex independent review 2026-07-07: APPROVE-WITH-CHANGES; all
five required changes incorporated below, revision markers `[R#]`)
**Owner directive (2026-07-07):** prioritize adoption/setup friendliness for less-technical
operators over new platform adapters. Docker requirement is an adoption barrier. Tier 3
single-process stays deferred (it solves multi-instance consolidation, not first-run).

## Goal

A semi-technical operator (can paste commands and follow a wizard; bounced off compose
profiles and layered env files) gets a working single-instance Discord bot with:

```bash
npx garbanzo-bot setup
```

No git clone, no Docker, no manual env editing. Wizard walks them through the Discord
developer portal, collects tokens/keys **and at least one channel to enable**, writes
config, starts the bot, and offers to install it as a service so it survives reboots.

Docker Compose remains the canonical full-stack path (monitoring, bridging,
multi-instance, Qdrant). This adds a second door; it does not move the first one.

## Grounding (codebase recon 2026-07-07, verified by independent review)

What already works bare:
- All runtime paths resolve from `PROJECT_ROOT` (`src/utils/config/index.ts:17-20`) via
  `path.resolve` — portable, no `/app` hardcoding. `data/`, `data/backups`,
  `baileys_auth/` auto-create. Env layering reads from `PROJECT_ROOT`
  (`applyEnvLayers`, `src/utils/config/shared.ts:43-46`).
- SQLite embedded; `VECTOR_STORE=none` gives keyword memory without Qdrant.
- Media shell-outs (ffmpeg/yt-dlp/piper/whisper) all fail soft — core chat runs without
  them.
- `scripts/setup.mjs` already has a `docker|native` deploy prompt; native branch is thin
  (prints only `npm run dev`, setup.mjs:803) and runs `npm install` (setup.mjs:237-247).
- Logging is stdout-only; signals limited to SIGINT/SIGTERM; loopback defaults for
  health/Ollama/bind hosts. `scripts/garbanzo.service` systemd unit exists but hardcodes
  checkout/node paths (garbanzo.service:8-9).

Hard blockers found (review-confirmed):
1. `config/groups.json` is read+parsed at module load with no try/catch
   (`src/core/groups-config.ts:28-29`), transitively imported by health middleware
   (`health.ts:27`) and persona (`persona.ts:8`) — missing/malformed file crashes every
   platform before startup. Committed in the repo so git-clone survives; any
   dist-only/npm install dies here.
2. `QDRANT_URL` schema default is `http://qdrant:6333` (`src/utils/config/vector.ts:7`) —
   resolves only on the compose network. **Neither docker-compose.yml nor the Helm
   chart pins it explicitly today** (compose relies on the schema default + depends_on;
   Helm sets it only when `qdrant.enabled` or an explicit URL — values default
   `enabled: false, url: ""`).
3. No npm publish surface: no `bin`, no `files`, `tsc` copies no assets (Dockerfile
   compensates: `config/`, `docs/PERSONA.md`, `docs/personas/`, `templates/`,
   `postgres-schema.sql`, Dockerfile:56-71). `dist/` alone is not runnable. Package is
   named `garbanzo` in BOTH package.json:2 and package-lock.json:2,8.
4. Config validation `process.exit(1)`s on missing/invalid env
   (`src/utils/config/index.ts:47-57`) and on zero providers (index.ts:91) — imported at
   module top-level by `src/index.ts:5`. Any CLI path that touches the runtime import
   graph before `.env` exists dies before it can help.
5. Discord channels are opt-in: missing `discord-channels.json` → `{channels: {}}`
   (`discord-config.ts:27`), unknown channels disabled (`discord-config.ts:58`),
   processor gates on `channelEnabled` (`processor.ts:215`). A quickstart that doesn't
   enable a channel yields a bot that starts and then ignores the entire server.

Easy fixes: wizard writes `COMPOSE_PROFILES` even for native; POSIX-isms
(`2>/dev/null`, `echo | pipe`) in vision/voice/links shell-outs break Windows media
features (fail-soft, core unaffected); `PIPER_BIN` hardcodes a Linux Homebrew path
(existsSync-guarded); `WHISPER_URL` default assumes a local sidecar (fails soft);
setup.mjs computes its root via `new URL(...).pathname` (not Windows-safe) unlike
config's `fileURLToPath`; setup writes `.bak` files not covered by `.gitignore`.

npm naming: `garbanzo` is a dead unpublished name (unreliable to claim);
**`garbanzo-bot` is available** (registry-checked 2026-07-07) and matches the repo.
Package: `garbanzo-bot`; installed command: `garbanzo`.

## Design decisions

### D1 — Home directory resolution (`GARBANZO_HOME`) [R1: heuristic replaced]

Mutable state (data/, baileys_auth/, config/*.json, PERSONA.md override, .env) must not
live inside the npm cache when running via npx. The original "repo checkout looks like
package.json + config/ + dist/" heuristic is WRONG — the published package ships exactly
those files, so npx would be misclassified as a checkout. Replaced with an explicit,
testable mode marker:

Resolution order:
1. `GARBANZO_HOME` env var, if set — always wins. The Dockerfile additionally sets
   `GARBANZO_HOME=/app` so container mode is explicit, not inferred.
2. **Packaged sentinel:** the publish build (and only it) writes `dist/.packaged`.
   If present → installed mode → home is `~/.garbanzo` (`os.homedir()`), created on
   demand. Repo builds (`npm run build`) and the Docker image (which builds from
   source in-repo) never contain the sentinel.
3. Otherwise → repo mode → `PROJECT_ROOT`, byte-identical to today.

Mode-resolution tests must cover all four topologies: source checkout, Docker `/app`
image layout, `npm i -g`, and npx cache install (simulated via `npm pack` + install
into a temp prefix).

Read-only assets (persona defaults, `docs/personas/`, config example templates,
`templates/5e-character-sheet.pdf`, `postgres-schema.sql`) ship inside the npm package
and are read from the package dir via `assetPath(...)`; mutable files resolve via
`homePath(...)`. Every current `resolve(PROJECT_ROOT, ...)` call site is classified into
one of the two — full inventory to sweep (each gets a mode test): db-schema.ts:26 (data),
whatsapp/connection.ts:21 (baileys_auth), outbound-safety.ts:96 (antiban state),
persona.ts:33-34, character/pdf.ts:16 (template), db-postgres.ts:200-203 (schema sql),
groups-config/bridge-map/rag-sources/discord-config JSON paths, openai-oauth.ts:30
(token), voice.ts:31 (voice models), plus Slack token state.

Env loading in installed mode: `.env` + `.env.<platform>` are read from `GARBANZO_HOME`
via the existing `applyEnvLayers` (baseDir becomes the resolved home).

### D2 — CLI surface (`bin: { garbanzo: dist/cli.js }`) [R3: import isolation]

- `garbanzo setup` — the wizard (see D3).
- `garbanzo start` — runs today's `dist/index.js` main.
- `garbanzo doctor` — environment report: Node version, resolved mode + home path,
  which config exists, which optional binaries are present (ffmpeg/yt-dlp), provider
  key presence (never values), health-port availability, current vs latest version
  (single registry request, fail-soft offline).
- `garbanzo service install|uninstall` — Linux systemd / macOS launchd. **Not valid
  from an ephemeral npx cache:** service install requires a durable install
  (`npm i -g garbanzo-bot`) and refuses under npx with exactly that instruction. The
  generated unit pins the resolved global bin path, `GARBANZO_HOME`, and the node
  binary; the existing `scripts/garbanzo.service` stays as the repo-mode template.
  Windows: prints Task Scheduler guidance, no automation this release.
- **Import isolation invariant:** `cli.js` must not touch the runtime config import
  graph (which `process.exit(1)`s without env) until after setup has written files —
  everything behind dynamic `import()` after arg parsing. Enforced by a test that runs
  `garbanzo setup --help` and `garbanzo doctor` in a clean temp home with NO `.env`
  and asserts exit 0.
- Updates: documented as `npx garbanzo-bot@latest` / `npm i -g garbanzo-bot@latest`;
  no self-updater this release.

### D3 — Wizard revamp (native-first quickstart flow) [R2: enabled channel required]

Extend `scripts/setup.mjs` (moves into the package as part of the CLI; root computed
via `fileURLToPath` for Windows safety):
- Platform: Discord is the promoted quickstart choice. WhatsApp stays available but the
  wizard flags the account-risk caveat (existing docs language). Slack/Teams not offered
  on the quickstart path (scaffold/dead-SDK).
- Discord walkthrough collects, in portal order: application (client) ID → bot token
  (with the create-app/enable-intents steps inline) → generated invite URL built from
  the client ID and the correct permission bits → owner user ID → **at least one
  channel ID to enable** (with the enable-developer-mode → right-click → Copy ID
  steps). The wizard writes `config/discord-channels.json` with that channel present
  AND enabled, so the first `@bot` mention works. Quickstart cannot complete with zero
  enabled channels.
- Defaults on the native path: `VECTOR_STORE=none` (with one line on upgrading to
  Qdrant later), monitoring off, `COMPOSE_PROFILES` NOT written. Writes
  `GARBANZO_HOME/.env` + `.env.discord` + `config/discord-channels.json` +
  `config/groups.json` (minimal valid).
- Packaged mode never runs `npm install` (deps are already vendored; the cache/prefix
  may be read-only) — the existing dep-install step is repo-mode only. Native next
  steps print `garbanzo start` (packaged) or `npm run build && npm start` (repo),
  never `npm run dev` as the primary.
- Ends with: start now? (foreground) / install service? (durable installs only, per
  D2) / where logs and data live.
- Docker path through the wizard is unchanged.
- Non-interactive mode gains native-Discord coverage: asserts the exact file set, no
  `COMPOSE_PROFILES`, `VECTOR_STORE=none`, enabled channel present (today's tests only
  dry-run Slack, tests/scripts.test.ts:69).

### D4 — Fail-soft first boot [R4: Helm + docs surface included]

- `groups-config.ts`: missing file → empty groups config + one warn log (same pattern
  as `discord-config.ts:38-43`); malformed → clear error naming the path, not a stack
  trace. Load via function, not at module import time.
- `QDRANT_URL` default → `http://127.0.0.1:6333`, with the compose hostname pinned
  explicitly where it belongs:
  - `docker-compose.yml`: add `QDRANT_URL=http://qdrant:6333` to BOTH bot services
    (it is NOT there today — verified).
  - Helm chart: template already emits `QDRANT_URL` when `qdrant.enabled` or
    `qdrant.url` is set (deployment.yaml:39); verify rendered value + add a helm test
    asserting the env var renders when enabled and is absent (thus loopback default)
    when not.
  - `.env.example` (currently pins `qdrant:6333` at ~:68) and `docs/CONFIGURATION.md`
    (states the old default at :97) updated; docs tripwire added banning the stale
    "default `http://qdrant:6333`" claim.
  - Net behavior change for compose/Helm users: none.
- `WHISPER_URL`/`PIPER_BIN`/`YT_DLP_BIN` documented as optional-feature envs in the
  reference; doctor reports their presence.

### D5 — Portability floor [R5: Windows claim downgraded]

- Convert vision/voice/links shell-outs from `exec("cmd 2>/dev/null")` strings to
  `execFile` with arg arrays (no shell, no POSIX redirects). Piper's `echo | pipe`
  becomes stdin writes. Correctness cleanup that also unbreaks Windows media.
- CI: add a matrix smoke job (windows-latest, macos-latest): `npm ci && npm run build`
  then boot with `HEALTH_ONLY=true` + assert `/health` 200 + clean SIGINT exit. Full
  suite stays ubuntu-only. Smoke is non-blocking (`continue-on-error`) this release.
- Supported tiers stated in docs, matching the CI evidence: Linux (full), macOS
  (supported, launchd), **Windows (experimental this release)** — the claim upgrades
  to "supported" only when the smoke job has been made blocking.

### D6 — Packaging & publish [R4: lockfile, tarball rehearsal, copy gate]

- Rename `garbanzo` → `garbanzo-bot` in `package.json` AND `package-lock.json` (root
  package name appears at lock:2 and :8; regenerate, don't hand-edit).
- `package.json`: `bin`, `files` (dist, shipped assets, LICENSE, README),
  asset-copy build step (config examples, personas, templates, postgres-schema.sql →
  shipped paths) + the `dist/.packaged` sentinel written by the publish build ONLY.
- **Pack rehearsal in CI runs the real path:** `npm pack` (which triggers the same
  prepack asset-copy+sentinel build as publish) → install tarball into a temp prefix →
  `garbanzo doctor` + `garbanzo setup --non-interactive` in a temp `GARBANZO_HOME` →
  assert boot with `HEALTH_ONLY=true`. Proves self-containment without publishing; do
  not rely on `prepublishOnly` alone.
- Publish automation: workflow job on release tag publishing `garbanzo-bot` with
  provenance. **Owner action required: create npm account/org and add `NPM_TOKEN`
  secret.** Until then the workflow skips with a notice.
- **Release-copy gate:** README/QUICKSTART must not promote `npx garbanzo-bot` until
  the first publish has actually succeeded — the docs land with the git-clone
  quickstart as primary wording plus a ready-to-flip npx section, and the flip happens
  (docs-only PR) after the owner adds the token and the publish job goes green.

## Non-goals (this release)

- Tier 3 single-process multi-runtime (deferred until a no-Docker user needs a second
  platform on one box).
- Standalone binaries (pkg / Node SEA) — npx path first; binaries only if Node-install
  friction proves real.
- Windows service automation; self-updater; hosted offering; Telegram (next release).

## Risks

- **R1 (highest):** T2 mode-resolution regressions breaking Docker/repo deployments.
  Mitigation: repo-mode branch is byte-identical behavior; explicit sentinel replaces
  inference; four-topology test matrix; full suite green; adversarial review focused
  here.
- **R2:** npx headline depends on owner-created NPM_TOKEN — handled by the D6
  release-copy gate (git-clone wording until first successful publish).
- **R3:** Windows CI flake — smoke non-blocking this release; docs say "experimental"
  accordingly (claim and evidence stay matched).
- **R4:** Wizard scope creep — the portal walkthrough is text + URL construction, no
  browser automation.
