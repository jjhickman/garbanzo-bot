## Workstreams

### WS1 — Telegram adapter (v3.3.0 headliner)

Stack: grammY, long polling (webhook mode deferred: long polling needs zero
inbound network config, which fits self-hosters). Dependency governance: the
owner named grammY in the directive; the implementation PR records the
license/maintenance check (MIT, active) and adds a Decisions Log entry, the
same way amqplib was approved.

Design mappings:
- **Privacy mode ↔ requireMention.** Telegram bots default to privacy mode
  (they see only commands/mentions/replies). Document as the recommended
  default and map onto the existing `requireMention` channel model; disabling
  privacy mode via BotFather gives full-group ingestion, equivalent to
  `requireMention: false`.
- **Config file:** `config/telegram-chats.json` following the
  discord-channels pattern (schema, example, gitignore, compose ro-mount,
  wizard writes one enabled chat).
- **Identity:** `TELEGRAM_BOT_TOKEN`, `TELEGRAM_OWNER_ID` (numeric), required
  iff platform=telegram (mirror the existing per-platform validation).
- **Inbound:** text, sender name, reply-to → quotedText, voice (OGG/Opus via
  getFile) → `InboundMessage.audio`, join events → welcome hook.
  **Credential rule:** Telegram file URLs embed the bot token
  (`api.telegram.org/file/bot<TOKEN>/...`) — they must never be persisted or
  logged raw (song_ideas stores `audio.url` verbatim today; logs must use
  the host-only discipline relay-capture already follows). The adapter
  downloads promptly and hands downstream code bytes or a `file_id`, never
  the token-bearing URL. A test asserts no token-bearing string reaches the
  database or the logger.
- **Outbound:** sendMessage with MarkdownV2 (new `buildFormattingInstruction`
  case + bridge `format-translate` case); MarkdownV2 escaping is the
  strictest of the platforms — property tests extended (R1).
- **Rate limits:** thin client-side throttle honoring Telegram's documented
  limits and 429 `retry_after` (relay-deliver gets a telegram case mirroring
  the Discord-only retry). No outbound-safety layer (official API, no ban
  risk).
- **Seam checklist:** config enum + `MessagingPlatform` union + envelope
  mirror + **bridge-map schema enum** + factory + runtime contract test,
  compose service (profile `telegram`, health port 3003, data volume — the
  compose contract test's pinned service list and volume set change
  explicitly), prometheus job telegram:3003, wizard menu + fields + BotFather
  walkthrough, doctor, PLATFORMS.md + README claims, bridge translate +
  relay-deliver, prompt-eval formatting case, helm values.platform.
- **Definition of done:** a live Telegram group runs the full command set and
  bridges to Discord in the owner's environment before tagging.

### WS3 — WhatsApp voice failure-path fix (v3.3.0, re-grounded)

Rev-1 premise was wrong; the review-corrected scope:
- Successful WhatsApp voice transcription already flows everywhere (the
  processor rewrites `inbound.text`). The real defect is the FAILURE path:
  Whisper down or download failed → the message is silently dropped before
  moderation, capture, and bridging.
- Fix: on transcription failure, continue processing with `[voice note]` as
  the text (a constant shared with relay-capture) instead of returning —
  moderation sees it, bridges relay the placeholder, recap/digest counts it.
  The AI-reply skip keys on a synthesized-placeholder FLAG on the inbound,
  not on text equality — a user who literally types "[voice note]" must not
  be silently ignored. On download failure, same. Tests: RED on today's
  drop behavior, then the continue path, plus the literal-text non-skip.
- No new transcription call sites; no gate unification (the two existing
  WHISPER behaviors — default-URL-always-try inline vs explicit-env-gated in
  Discord-origin capture — stay as they are and get documented in
  CONFIGURATION.md; unifying them is a behavior change nobody asked for).

### WS4 — Community operations web UI (phase 1 in v3.3.0 by owner decision;
### phases 2–3 in v3.4.0–v3.5.0)

The village model gives the information architecture; the existing `/admin`
server gives the mount point and token auth.

- **Phase 1 (v3.3.0): read-only.** Sections: Overview (connection, uptime,
  instance identity), Memory (browse facts: text, category, source tag,
  shared status), Bridges (route map + outbox/dead-letter counts), Health
  (inline key metrics, link to Grafana). Server-rendered HTML like the
  current admin page — no SPA, no build step, no new dependencies.
- **Write-gate design task (v3.3.0, doc-only):** the container-aware
  mutation-gate design (candidate mechanisms per the security invariants
  below) is produced during the v3.3.0 cycle so phase 2 implementation can
  start immediately after.
- **Phase 2 (v3.4.0): safe mutations.** Memory curation (delete/share/unshare
  — parity with `!memory`), charter view/edit (persona file, group/channel
  toggles), config export/import (download an archive of GARBANZO_HOME
  config; upload with diff preview + automatic .bak, reusing the wizard's
  preserve/merge semantics). Drag-and-drop + file browsing = plain HTML5
  input + drop zone.
- **Phase 3 (v3.5.0):** bridge-map editor validated against BridgeMapSchema;
  demand-proven follow-ons.
- **Security invariants (settled INTENT; mechanism designed at phase 2):**
  reads require MONITORING_TOKEN as today; mutations are gated beyond the
  read token; every mutation is audit-logged (what, when, from where);
  destructive actions take a confirm step; config writes pass the same
  schema validation as boot. Phase 1 ships no mutation endpoints at all.
  The gate MECHANISM is a phase-2 design task with container awareness:
  a naive loopback-source check breaks under Docker (host requests reach
  the container from the docker network gateway, not 127.0.0.1, and with
  the userland proxy LAN traffic can share that source) — candidate
  mechanisms to evaluate there: a separate admin listener bound only
  inside the container namespace / to an operator-chosen interface, a
  distinct write-token tier above MONITORING_TOKEN, or explicit
  trusted-source handling with documented compose behavior. The realistic
  WhatsApp deployment (health port published on 0.0.0.0) is the design's
  reference threat shape.
- **Labels (owner-decided, Q3):** shipped UI uses PLAIN primary labels
  ("Memory", "Bridges", "Health"); "Lore" appears as the promoted alias
  where memory is introduced ("Memory — your community's lore").
  "Watchtower"/"village" language stays in docs and landing copy, not in
  the shipped UI.

### WS10 — Persona gallery + wizard persona picker (v3.3.0, owner-directed)

Six shipped personas alongside the Garbanzo Bean default, each a working
demonstration of a real feature set, living as markdown files in
`docs/personas/` (the persona loader already derives name + emoji from the
heading, so the gallery is zero-code content):

- **Riff 🎸** — bands and music projects (band features: songs, rehearsals,
  setlists, idea capture).
- **Quill 🎲** — tabletop groups (memory-as-canon, session recaps,
  scheduling, the character-sheet generator).
- **Margie 📚** — book clubs (book lookup, reading schedules, spoiler-aware
  moderation, who-recommended-what memory).
- **Bea 🏡** — neighborhood and mutual-aid groups (welcomes, events,
  weather, practical community memory).
- **Patch 🔧** — open-source and maker communities (contributor welcomes,
  decision memory, digests, code-of-conduct moderation).
- **Callie 🎭** — theater/dance/rehearsal-based groups (demonstrates that
  the band features generalize to any practice group).

Gallery files live in `docs/personas/gallery/` — a subdirectory, because
`docs/personas/*.md` filenames are PLATFORM KEYS the loader resolves by
`MESSAGING_PLATFORM`; gallery names must not share that namespace.

Wizard picker: the interactive flow offers the gallery (name + one-line
use case) plus "custom persona file" and "default"; non-interactive gains
`--persona=<gallery-name|path>`. **Copy destination (review-mandated):**
the selection is written to the PLATFORM-KEYED home slot,
`GARBANZO_HOME/docs/personas/<platform>.md` (`.bak` on overwrite) — NOT
home `docs/PERSONA.md`, which the persona loader's precedence
(`persona.ts:44-55`: platform file beats PERSONA.md, and the SHIPPED
`docs/personas/discord.md` shadows a home PERSONA.md) would silently
ignore on Discord. WS4 phase 2's charter editing targets the same slot.
Picking Riff or Callie offers to enable `BAND_FEATURES_ENABLED` in the
same run. PLATFORMS/CUSTOMIZATION docs get a gallery section with the
one-line use cases (feature claims limited to what exists — lookup_book,
band tables, character sheets, recaps are all real tools today).

Test-pin contract: the v3.2 wizard tests pin the exact non-interactive
file set — the default run WITHOUT `--persona` stays byte-identical
(existing pins prove it), and a new pinned case covers `--persona=<name>`
including the copied file at the platform-keyed path. The npm package
already ships `docs/personas/` wholesale, so the gallery rides into the
tarball; the pack-rehearsal file count changes and is updated knowingly.

Voice note: persona files are product content and are SUPPOSED to have a
voice — the plain-register rule governs marketing copy and operational
surfaces, not persona personality. Persona names never name infrastructure
(standing decision); each file states its target use case in a comment so
operators understand it's a starting point to edit.
